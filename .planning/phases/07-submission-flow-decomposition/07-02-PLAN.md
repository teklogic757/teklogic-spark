<objective>
Separate persistence from side effects so valid ideas are not coupled to outbound notification success.

Purpose: The current submit actions still interleave core database writes, point updates, and best-effort email dispatch in one broad routine. This plan introduces an explicit post-persistence boundary so the system can save the idea first, then perform notifications and other non-critical work in a clearly isolated, best-effort path.
Output: Shared persistence and post-persist helper functions that make the success/failure boundary explicit and resilient.
</objective>
