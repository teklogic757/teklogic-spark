<objective>
Create the durable limiter foundation Phase 8 depends on by replacing restart-prone process memory with a shared database-backed limiter contract.

Purpose: The current `Map`-based limiter resets on deploys and on every new process, which makes protection unreliable in the exact deployment shapes the app is moving toward. This plan introduces a Supabase/Postgres-backed limiter path that matches the project's modest scale without adding Redis infrastructure.
Output: A migration that defines the durable limiter store plus an async `rate-limiter` module that all current call sites can reuse.
</objective>
